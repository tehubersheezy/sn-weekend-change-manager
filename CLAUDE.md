## Commands

- `npm run build` — plain `now-sdk build`; the prebuild generates the stylesheet itself, so this and a bare `now-sdk build` are the same command. Must pass before deploy.
- `npm run deploy` — build, assert the deploy shape, then `now-sdk install`. **Never deploy unless Abey explicitly asks.** **Stop the dev server first** — see "Dev server vs deployed build".
- `npm run dev` — local dev server with LiveReload + `/api`,`/amb` proxies (Abey usually runs it himself; check ports 3000/8081 before starting another). Read "Dev server vs deployed build" below before mixing it with builds or deploys.
- `npx tailwindcss -i ./src/client/styles/theme.css -o ./src/client/styles/tailwind.generated.css --watch=always` — CSS watcher; plain `--watch` exits when stdin closes in background shells.
- `npx tsc --noEmit -p src/client/tsconfig.json` — the standard check after edits.
- Instance queries: **always via the `sn` skill** (Abey's standing preference). Raw `npx now-sdk query` caps at 100 rows/page and has no aggregates — build/deploy/auth are the only now-sdk jobs.
- **Verifying a deployed scripted REST route: curl the dev proxy.** `curl "http://localhost:3000/api/x_912401_weekend_c/jira/issue?key=NET-4451"` authenticates straight through to dev421992 — no credentials to dig up, no `sn` round-trip, and it returns the real body. The fastest way to prove a route actually works after `npm run deploy`, rather than trusting "Installation completed".

## Dev server vs deployed build (hard-won 2026-07-11)

Both builders write the **same `dist/static`**, in two different shapes, and the dev server serves whatever is there:

- **Deploy shape** (`npm run build`): `index.html` references `/uxasset/externals/x_912401_weekend_c/main.jsdbx?uxpcb=$[Jelly]`, and the whole file is XML-serialized — self-closing `<script/>` and `<div id="root"/>`. Only valid after the instance processes it. A raw browser parse treats `<script/>` as an *unclosed* tag and swallows the rest of the document as script text: no `#root`, **blank page, zero console errors**. That silhouette = deploy-shaped HTML on the dev server.
- **Dev shape** (dev server's own watch build): `index.html` references local `main.js`, tags properly closed.
- The watch build re-fires on **source changes only**. Running `npm run build`/`npm run deploy` while the dev server is up leaves deploy-shaped files in `dist/static` — afterwards, `touch` any client source file to flip it back to dev shape.
- `/uxasset/externals` (like `/api`, `/amb`, `/uxta`) proxies to dev421992. A bundle path missing locally falls through to the proxy, so the dev page can silently run the **deployed** bundle — local edits appear ignored. Diagnose by checking the served root's `<script src>`: `main.js` = local build, `main.jsdbx` = instance code via proxy.
- Ports: the dev server binds **localhost:3000** (ignores `PORT`, no CLI flag; the 8081 in older notes was a second instance falling back because 3000 was occupied). 35729 = LiveReload, 3001 = internal WebSocket.
- **Never deploy with the dev server running (hard-won 2026-07-13).** The build regenerates `styles/tailwind.generated.css`, which lives *inside* the directory the dev server watches — so a deploy build wakes the watcher, which rebuilds into the same `dist/static` the installer is about to upload. `now-sdk install` then finds `main.js` and `main.jsdbx` both claiming one `sys_ux_lib_asset` sys_id, prints "the last entry from each conflict will take precedence", **and exits 0** — a green "Installation completed" that may have just put the DEV bundle in the record the deployed page loads. `npm run deploy` now hard-fails on this (`scripts/assert-deploy-shape.mjs`), but the rule is: stop the dev server, then deploy, then restart it.
- Deploy hygiene: `npm run deploy` rebuilds before installing, so the old "remember to build first" rule is now enforced rather than remembered. Install still uploads the built artifact without rebuilding, so never call `now-sdk install` directly.
- `src/client/main.tsx` carries dev-resilience that is inert on-instance: it recreates `#root` when missing (only ever true under the swallowed-body parse above) and prunes unversioned duplicate app CSS only when the authoritative `?uxpcb=` copy exists (always true deployed; in dev the bundler-injected `&v=`-stamped link is the only stylesheet and must survive).

## Architecture

- `src/fluent/` — UiPage definition (endpoint `x_912401_weekend_c_console.do`, `$id` key is still `Now.ID['incident-manager-page']` — intentional, do not change) + application menu.
- `src/client/app.tsx` — shell: TopNav (screen menu Plan/Execute/Review, world clocks HK/Zug/NY, Live dot, window badge), full-width header (phase headline, window controls, compact stat chips), then a 50/50 split: list pane (toolbar row = state tabs + group/assignee selects + List/Timeline toggle) and detail pane. URL params: `?screen=`, `?id=`, `?jira=` — Polaris iframe bridging in `pushUrl`. The detail pane is a three-way switch: a Jira issue OUTRANKS the change behind it (`?jira=` layers OVER `?id=` rather than replacing it, so closing the issue returns to the change it was opened from, on its Jiras tab), and with neither set the pane rests on the activity feed.
- `src/client/utils/phases.ts` — screen/state model. Plan={-4,-3,-2}, Execute={-1,-2,0,3} (keeps completed work visible), Review={0,3}. Default screen by time vs window.
- `src/client/utils/weekendWindow.ts` + `datetime.ts` — window math: Fri `startTime` → Sun `endTime` in a user-selected IANA timezone (`WindowConfig.timeZone`, default the browser zone; times default 17:00→23:59, all configurable via the Settings dialog, persisted in localStorage `wcm.windowConfig`), week offset navigation, all Intl-based zone↔UTC (never hardcode offsets; floor to whole seconds before offset math). The zone reaches display components via `context/TimeZone` (`useTimeZone()`) — it MUST appear in every date-deriving `useMemo`'s deps (the feed day-grouping is the trap); `datetime.ts` caches one `Intl` formatter set per zone, and the window label's old ` ET` suffix is now a live `zoneAbbreviation`. The same dialog also persists LLM endpoint/token to `wcm.llmConfig` (plumbing only, no consumer yet).
- `src/client/hooks/useWeekendChanges.ts` — data owner: Table API load + silent AMB refetch.
- `src/client/services/ActivityService.ts` + `hooks/useActivityFeed.ts` + `components/ActivityFeed.tsx` — the detail pane's resting view: live weekend activity feed (comments + work notes from `sys_journal_field`, state/approval transitions from `sys_audit`), scoped by chunked sys_id IN-lists over the loaded window population. No AMB channel of its own — every feed-worthy event also updates the parent record, so it refetches off `changes`/`tasks` identity changes. Feed hook lives in app.tsx so back-nav from a detail is instant. `utils/stateLabels.ts` maps audit's raw codes to labels.
- **Jira (foreign records)** — `src/server/jira.ts` + `src/fluent/jira.now.ts` expose two scoped REST routes: `GET /jira/issues?keys=A,B` (batch summaries, for lists/badges) and `GET /jira/issue?key=A` (one issue + its references). Client side: `services/JiraService.ts`, `hooks/useJiraSummaries.ts`, `components/JiraDetailView.tsx`, `JiraBadges.tsx`, `JiraList.tsx`. **The payload has two halves from two systems and they are siblings, never nested.** `issue` is MOCK — there is no Jira on this instance, so it's 20 hand-authored fixtures in `src/server/jira.ts`, one per key the seeded change tasks carry, with statuses that track their tasks. `references` is REAL — a live `GlideRecordSecure` query on `change_task.correlation_display` joined to the parent change, i.e. "which weekend work depends on this issue", the one question the real Jira page cannot answer and the reason the surface exists. An unknown key is a 200 with `issue: null` and its references intact, not an error. Wiring a real Jira = replace `findIssue()` with an `sn_ws.RESTMessageV2` callout; nothing else changes, client included.
- `src/client/services/SnowAmb.ts` + `hooks/useAmb.ts` — Bayeux/CometD WebSocket to `/amb` (record watchers, reconnect/resubscribe). **Only works deployed on-instance** (cookie auth on WS upgrade); localhost always shows "Connecting". Protocol reference: `~/Projects/mlx-audio/servicenow-amb/protocol.md`.
- `src/client/components/ui/` — vendored shadcn components (relative imports only, no `@/` aliases), themed via `styles/theme.css` tokens.
- `src/client/components/TimelineView.tsx` — Gantt view; its bar palette is dataviz-validated against the cream canvas — don't swap colors without re-running the validator.

## ServiceNow gotchas (hard-won, trust these)

- **`change_request` planned dates are `start_date`/`end_date`** (labels say "Planned start/end date"). `planned_start_date` does NOT exist on change_request — and ServiceNow silently drops invalid encoded-query terms, returning the whole table. `change_task` DOES use `planned_start_date`/`planned_end_date`.
- change_request states: New=-5, Assess=-4, Authorize=-3, Scheduled=-2, Implement=-1, Review=0, Closed=3, Canceled=4. change_task: Pending=-5, Open=1, In Progress=2, Closed=3. Window query excludes New and Canceled.
- State isn't writable at insert (change model BRs); seeding requires stepping transitions (`approval=approved` to pass Assess) or temporarily disabling the state-model BRs (re-enable after!). `type` derives from `chg_model` at insert (Standard `e55d0bfec343101035ae3f52c1d3ae49`, Emergency `62d10fa1c303101035ae3f52c1d3aec1`).
- `assigned_to` must be a member of `assignment_group` (403 otherwise).
- Affected CIs = `task_ci` table (task, ci_item); writes blocked past New state by "Change Model: Read only CI" BRs (same disable/re-enable trick).
- Table API: always `X-UserToken: window.g_ck` + `sysparm_display_value=all` (fields arrive `{value, display_value}` — `utils/fields.ts` helpers).
- **URI length limit**: the instance front proxy (snow_adc) 414s at **8192 bytes of the FULL request URI** — the path and every other `sysparm_*` share that budget, so it is not a query-length limit (measured 2026-07-14: 245 bare sys_ids in an IN-list = 8188 B OK; 246 = 8221 B → `414 Request-URI Too Large`; the same 245 ids plus 200 bytes of unrelated padding also 414s). A full window's 350 sys_ids in one IN-list breaks. Chunk IN-lists at 80 ids (~2.6KB, verified OK) — `services/tableApi.ts` `chunkIds`.
- `sys_journal_field` keys by `element_id`, `sys_audit` by `documentkey` — both plain strings, NOT references (no dot-walking; IN-list scoping only). sys_audit stores raw codes for choice fields and sys_ids for references (no display values), and approval-history journal noise mirrors into it as "JOURNAL FIELD ADDITION" rows — whitelist `fieldname` when querying. `user`/`sys_created_by` are username strings; resolve via sys_user (and guard for accounts with empty `name`). Both tables read fine as admin; non-admin ACLs untested.
- `src/client/index.html` invariants: inline CDATA `Array.from` polyfill before the module script, `<sdk:now-ux-globals>`, `?uxpcb=$[UxFrameworkScriptables.getFlushTimestamp()]` on the module src. Breaking any of these breaks Polaris silently.
- **`src/server/*.ts` stays in a conservative ES5 dialect** — no arrow functions, template literals, spread, destructuring, `.includes()`, `for…of`. Match `activity.ts` exactly. ServiceNow's server engine is not guaranteed to be the modern one, the tsconfig's `target: es2022` is a typecheck setting and proves nothing about the runtime, and a syntax error here **would not surface until install**. There is no local harness that catches it.
- **A server route must return RAW choice codes, not display labels.** `StateBadge` switches on the numeric value (`'-1'`, `'3'`); `TaskStateBadge` happens to match on the label, so a label "works" there and silently falls through to the neutral grey pill on `StateBadge` — right text, wrong colour, no error, ships. Send `getValue('state')`, never `getDisplayValue('state')`, and let `utils/stateLabels.ts` `asStateField()` do the code→label→variant mapping client-side (the same thing `sys_audit` already forced).

## Frontend gotchas

- **The bundler ships bare `#subpath` imports to the browser — blank app in dev AND deployed (hard-won 2026-07-16).** `@servicenow/isomorphic-rollup`'s vendored node-resolve cannot resolve Node subpath imports (a package.json `imports` field, `#`-prefixed specifiers): it resolves the right file, then its containment check compares against a doubled-slash package URL (`outside package //Users/…`), always fails, and rollup's fallback marks the specifier **external** — a mere warning in the build log ("treating it as an external dependency") and a module-level `import{minpath}from"#minpath"` in main.js/main.jsdbx. The browser throws `Failed to resolve module specifier "#minpath"` before any app code runs. First tripped by react-markdown → vfile; found it had ALSO been deployed broken. Fix: `patches/vfile+6.0.3.patch` (patch-package, `postinstall`) rewrites vfile's three `#min*` imports to their `.browser.js` targets — correct because this bundle only ever targets browsers. Any future dep using `#` imports hits the same wall: the build-log warning is the tell, and the gate is `grep 'from"#' dist/static/main.js` after adding a dependency. (The `node:fs` warnings from @anthropic-ai/sdk are benign — tree-shaken out of the final bundle.)
- **An unknown Tailwind class emits NO CSS, silently.** No typecheck error, no build warning, no design-hook complaint — the element just renders unstyled. This bites hardest because **DESIGN.md token names are not Tailwind class names**: it says `{colors.canvas}` and `{colors.hairline}`, but the classes are `bg-background` and `border-border` (there is no `bg-canvas`, no `border-hairline`). Writing the token name gives you an invisible element. `src/client/styles/tailwind.generated.css` is the **only artifact that knows the truth** — after any styling work, grep it for the classes you wrote. Both gates in this repo (tsc, the impeccable hook) are blind to this.
- **After install, the instance keeps serving the PREVIOUS deploy's CSS for a few minutes — a stale fetch right after `npm run deploy` is not a failed deploy.** The bundle pulls its stylesheet from `/uxta/<sys_id>.assetx?path=styles/tailwind.generated.css&v=<stamp>`, served `cache-control: public` with a year-long expiry. Observed twice on 2026-07-12: install completes, the new bundle ships with a **new** `v=` stamp (it does track content — `905b781e` → `2311a94d` when the CSS changed), and yet the endpoint returns the *old* CSS bytes at that new URL for some minutes before catching up on its own. Cache-busting the request (`&cb=1`) does not reliably force it. So: don't diagnose off a fetch taken seconds after install, and don't conclude the CSS "didn't upload". Verify by curling the `.assetx` URL through the dev proxy after a pause — but note the failure mode this masks is genuinely nasty when it *is* real (new JS with new class names, old CSS with no rules for them), because the JS is immune: `index.html`'s `?uxpcb=$[UxFrameworkScriptables.getFlushTimestamp()]` re-stamps on every install.
- **In Tailwind v4, `translate-*` is NOT `transform` — a keyframe that animates `transform` COMPOSES with it instead of overriding it (hard-won 2026-07-13).** `-translate-x-1/2` compiles to the standalone `translate` property; CSS applies `translate`, then `rotate`, then `scale`, then `transform`. So `@keyframes dialog-in` re-stating `translate(-50%, -50%)` inside its `transform` — correct and necessary in v3, where an animation's `transform` *did* clobber the utility — displaced the dialog **100% of its own size**, and `fill-mode: both` held it there. Symptom: the panel's bottom-right corner is pinned to the centre of the viewport (it sits half off the left edge, header above the top of the window, and a taller record only pushes it further out) — which reads exactly like a broken build, and sent a whole session chasing the build system. Keyframes animate scale/opacity; the utilities own the centring. Same trap awaits any `animate-*` keyframe touching `transform` on an element that also carries a `translate-*`/`scale-*`/`rotate-*` class.
- A `fixed` element's `w-full` is the **viewport**, so `max-w-*` is the only thing holding a dialog off the screen edges; on a window narrower than the cap the gutter is zero. `DialogContent` clamps with `w-[calc(100%-2rem)]` — on *width*, because callers override `max-w-*` and tailwind-merge would drop a competing `max-w` from the base line.
- Theme tokens live in `theme.css`'s `@theme inline` block — that block is the authoritative list of what `bg-*`/`text-*`/`border-*` names actually exist. Check it before inventing a class.
- `lib/utils.ts` `TEXT_SCALE` must stay in sync with the `--text-*` block in `theme.css`, or tailwind-merge misclassifies a size token as a color and `cn()` silently drops it.
- **The two builders disagree about CSS, so grep the SHIPPED css (`dist/static/styles/`), not just the generated one.** Hard-won 2026-07-12: Tailwind v4 emits `hover:` double-nested (`.hover\:bg-x { &:hover { @media (hover: hover) { … } } }`). The **dev server's esbuild** hoists that inner `@media` to the top level and **loses the parent selector**, shipping an orphaned `@media (hover: hover) { &:hover { … } }` that matches nothing — every `hover:`/`group-hover:` class is dead CSS *on localhost*. The **deploy build (rollup) lowers the same input correctly** (verified by building both pipelines with and without the fix), so this was a **dev-only** breakage; the deployed page was never affected. Diagnostic trap: `tailwind.generated.css` looked *correct*, so the standard "grep the generated CSS" gate passed — and grepping for the bare token name (`hover-ink`) matches the `:root` *declaration* even when no selector exists. Use an anchored selector grep (`^\s*\.hover\\`) against **both** files. The bug is the nesting **order**, not nesting itself: fixed by redefining the variant in `theme.css` with `@media` on the OUTSIDE of `&:hover` (`@custom-variant hover { @media (hover: hover) { &:hover { @slot } } }`), after which esbuild hoists the `@media` and the parent selector survives. The touch gate is fully preserved — do NOT "simplify" this to `@custom-variant hover (&:hover)`, which also fixes dev but by deleting the gate from production. `active:`/`focus-visible:`/`sm:` were never affected (never double-nested — which is why press/focus states always worked, and why this stayed latent until the app's first-ever `hover:` class in 2026-07-12's motion pass).

## Working conventions

- **No monospace, ever** (global preference) — sans stack everywhere including numbers, IDs, code/plan panels. `tabular-nums` on the sans face is the tool when digits must align.
- Theme token classes only in TSX (bg-primary, text-ink, bg-surface-card…) — no raw hex outside `theme.css`/validated chart palettes. See Frontend gotchas: a class that isn't real fails silently.
- DESIGN.md rules: serif display 400 with negative tracking for headlines only; coral scarce; hairline borders; no shadows. Hover = warm hue shift via the `--hover-*` tokens (Abey-sanctioned amendment 2026-07-12 — see DESIGN.md "Interaction: Hover & Motion"; Jira clickables hover in-family via `--jira-hairline-hover`, never toward coral; Radix highlight states still use surface tones).
- **DESIGN.md > Foreign Records** governs anything Jira. The one-line version: **blue is a GROUND, never an ACCENT** — it marks provenance ("this record does not live in ServiceNow"), and the moment it's used to make something *stand out*, the rule is broken. It may never paint a link, a button, a status on a native record, or a chart series; coral is still the system's only voltage, and the "no cool blue as a brand accent" Don't stands unamended. ServiceNow badges are PILLS (hue-coded); Jira badges are STAMPS (`rounded-xs`, one hue, three densities), so a Jira "Done" cannot be confused with a ServiceNow "Closed". The Jira family has ONE chromatic token (`--jira-ink`) and no light fill — that is arithmetic, not an oversight (a blue that clears 3:1 collapses into its own ink step; read the section before trying to "complete" the family).
- Do not deploy or run browser verification unless explicitly asked. Typecheck is the default gate.
- Data seeding/instance mutations are usually delegated to a background agent; report BR disable/re-enable honestly.
